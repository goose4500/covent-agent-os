Covent Agent OS diagrams
Post-rebuild architecture · single-file guide

10 architecture diagrams for explaining how Covent Agent OS works.

A team-facing single-file guide for the post-rebuild Slack ↔ Pi ↔ Linear system. The narrative starts with value flow, then drills into runtime, tools, artifacts, subagents, approvals, and deployment.

Human / Slack surface Pi / runtime Durable truth / artifact Caveat / risk boundary

Core thesis

Slack selects an Action. Pi executes with default-on app-approved capabilities. Artifacts return to Slack/Canvas/Linear. Durable truth lives in Linear, GitHub, and repo docs.

Route prefixes shape workflow instructions. They are not tool allowlists or security sandboxes.

Railway deployment / runtime topology

Gives operators the production mental model: one Socket Mode worker, env-only secrets, persistent Pi auth/session state, and canary rollback.

Deployment diagram
Slack appSocket Mode, Assistant, app_mention, DMs, /thread-spec
Railway covent-pi-momBun worker, no public ingress, auto-deploy from main
PI_AGENT_DIR volumeauth.json seed, session continuity, thread map
External APIsSlack Web API, Linear GraphQL, model/web providers
Child pi CLIrequired for subagent runs
Canary / rollbackcovent-pi-mom-v2 hot target; avoid split-brain workers
Ops rule
Verify Railway variable keys only; never paste values. Do not run duplicate workers unless intentionally debugging.
