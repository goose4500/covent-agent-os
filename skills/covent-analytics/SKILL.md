# COVENT-ANALYTICS

Covent analytics is the R&D and implementation track for turning `Dispo-Genius/covent-frontend` into a governed, first-party growth intelligence system rather than a scattered click-tracking setup. The core strategy is to evaluate Amplitude vs Mixpanel from first principles, but avoid making either vendor the architecture: feature code should eventually call a Covent-owned analytics wrapper with typed events, attribution, identity/account context, PII guardrails, and pluggable adapters for Amplitude, Mixpanel, no-op/test, and future server/warehouse ingestion. The main specs live in `/home/jfloyd/covent-source/covent-frontend/specs/Analytics/amplitude-vs-mixpanel-rnd.md` for the strategic R&D/architecture decision and `/home/jfloyd/covent-source/covent-frontend/specs/Analytics/amplitude-modern-sdk-implementation.md` for the Amplitude implementation plan.

Primary code boundaries to inspect before making changes: current Mixpanel wrapper at `src/features/(shared)/Analytics/lib/mixpanel.ts`, event constants/types at `src/features/(shared)/Analytics/lib/constants.ts` and `types.ts`, init hook at `src/features/(shared)/Analytics/lib/hooks/useInitializeMixpanel.ts`, initializer component at `src/features/(shared)/Analytics/components/AnalyticsInit/AnalyticsInit.tsx`, product mount point at `src/components/Layout/AppProviders/AppProviders.tsx`, marketing layout gap at `src/app/(marketing)/layout.tsx`, Mixpanel proxy rewrites in `next.config.ts`, marketing emitter at `src/features/Marketing/lib/hooks/useMarketingEvents.ts`, and stub server routes at `src/app/api/marketing/events/route.ts`, `src/app/api/marketing/leads/route.ts`, and `src/app/api/marketing/qualify/route.ts`. The intended interaction boundary is: product/marketing feature code → `Analytics/lib/analytics.ts` wrapper → common context/attribution/identity/event contract → vendor/server adapters; do not add raw SDK calls directly inside feature components.
