---
name: linear-subissue-audit
description: Use this skill whenever the user asks to go through, review, audit, summarize, or inspect sub-issues under a Linear parent issue, especially when they provide a parent title plus project/team context. This skill exists to avoid slow or context-heavy Linear searches and should be used even if the user does not explicitly say “audit.”
---

For a Linear parent/sub-issue audit, use the `mcp` Linear tools in this order: first `linear_list_issues` scoped by `team`, `project`, `limit: 250`, `includeArchived: true`, and `orderBy: "updatedAt"` rather than relying only on `query`, because newly-created issues or exact-title searches can lag or miss results; page with `cursor` if `hasNextPage` is true. Find the exact parent by title or identifier, then call `linear_get_issue` with `includeRelations: true`, then `linear_list_issues` with `parentId` set to the parent identifier (for example `FE-457`) to get direct sub-issues.

After the child list is known, batch detail reads in parallel when available: use `multi_tool_use.parallel` for `linear_get_issue(includeRelations: true)` and `linear_list_comments` for each child; otherwise do those calls sequentially. Security guard: if the child-list snippet already contains credential-looking material (`xox`, `ghp_`, API key/token/secret/password/private key), do not fetch that issue’s full body or comments unless the user explicitly asks; report “sensitive credential material present” and recommend redaction/rotation. Never mutate Linear unless explicitly asked, and never print credential-looking secrets from issue bodies. Return a compact Markdown report with parent metadata, a summary table, per-sub-issue status/assignee/requirements/comments/blockers/next action, and gaps or cleanup recommendations; if no filesystem write tool is available, return the intended file contents inline so the parent/harness can save them.
